1. RACE — nouveaux champs
sqlRACE :
  + pourcentage_vavy   INT  -- ex: 60
  + pourcentage_lahy   INT  -- ex: 40
  + duree_incubation   INT  -- ex: 21 jours
  + capacite_pondation INT  -- ex: 40 oeufs/vie
```

**Pourquoi stocker les deux** (`pourcentage_vavy` ET `pourcentage_lahy`) même si 60+40=100 ?
```
→ Données fixes de la race, pas calculées
→ Pas de risque d'erreur de calcul (100 - vavy)
→ Plus lisible dans les requêtes SQL
→ Flexible si un jour vavy + lahy ≠ 100
   (ex: 5% mort-nés non comptabilisés)
→ Cohérent avec capacite_pondation et duree_incubation
   qui sont aussi des données directes

2. LOT — nouveau champ sexe
sqlLOT :
  + sexe        ENUM('vavy', 'lahy', 'mixte')
  + pourcentage INT  -- ex: 60 (% de ce sexe dans le lot)
```

**Formulaire création lot :**
```
Champ existant : race, date_entree, nombre_initial...

Nouveau :
  Sexe       : [select] vavy | lahy | mixte
  Pourcentage: [input]  ex: 60%
               → pré-rempli automatiquement depuis la race
               → modifiable manuellement
```

**Exemple :**
```
Lot source (pondeuses) → sexe: vavy, pourcentage: 100%
Nouveau lot après incubation → sexe: mixte, pourcentage: 60% vavy

nb_lamokany    = nombre_oeufs × pourcentage_lamokany / 100

### 5. SITUATION — nouveaux champs
```
AVANT :
  Nb atody | Prix atody

```

**Calcul :**
```
nb_atody_valide  = SUM(recensements) - SUM(lamokany)
nb_lamokany      = SUM(nombre_oeufs × pourcentage_lamokany / 100)

prix_atody_valide = nb_atody_valide × prix_unitaire_atody
prix_lamokany     = nb_lamokany     × prix_unitaire_atody

benefice += prix_atody_valide + prix_lamokany
```
